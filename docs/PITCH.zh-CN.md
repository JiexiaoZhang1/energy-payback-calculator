# 项目演说稿

这些稿子可以直接用于课堂展示、作品集讲解、面试、录屏介绍或项目路演，也可以根据现场时间删减。文案只说明项目实际完成的功能，不把估算结果说成保证收益。

## 30 秒版本

这个项目是一个家庭用电回本计算器。用户可以输入一天里不同时段的电价，选择家里的可调度设备，比如储能电池、热泵热水、电动车充电和可预约家电，再填写设备成本。系统会按照“便宜时段多用电或充电，贵时段少从电网买电”的思路，估算每类设备一年大概能省多少钱，以及整体大概几年能回本。前端负责完整交互和本地兜底计算，Express 后端负责计算 API 和保存分享方案。

## 90 秒版本

现在很多地方的电价不再是全天固定价格，而是越来越依赖时间。比如凌晨便宜，傍晚高峰昂贵，有些市场甚至会出现几个小时的负电价。对有储能、电动车、热泵或可预约家电的家庭来说，这个价差会影响实际电费。

家庭用电回本计算器把这个问题做成了一个简单的网页应用。用户先设置一天里的电价时段，包括负价、低价和高价。然后选择家里有哪些可以灵活安排用电的设备，比如储能电池、热泵热水、电动车充电和可预约家电。用户还可以调整设备成本和效率参数。

计算后，页面会清楚展示三件事：预计多久回本、每年大概省多少钱、每类设备分别贡献多少节省。结果页还会给出简单的每日用电建议，告诉用户哪些时段更适合用电或充电，哪些高价时段应该尽量减少从电网买电。

前端使用 React、TypeScript、Vite、Framer Motion 和 Lucide 图标。小型 Express 后端提供 API 计算和方案保存。如果后端不可用，浏览器会运行同一套本地计算，因此 GitHub Pages 上的静态演示仍然能使用。

这个项目的目标不是精准预测每个家庭的真实电费账单，而是把电价差、可移动用电、年节省和设备回本之间的关系讲清楚，给用户一个可以用于初步比较的估算结果。

## 3 分钟版本

今天我展示的是一个叫家庭用电回本计算器的网页应用。它是一个小型全栈项目，重点是估算家庭在分时电价下调整用电时间后，大概能省多少钱，以及设备投入多久能回本。

项目背景是很多电价已经不是全天一个固定价格。一个家庭可能会遇到凌晨电价低、白天正常、傍晚高峰昂贵的情况。有些市场甚至会在部分时段出现负电价。对拥有可调度设备的家庭来说，这就带来一个很实际的问题：如果把一部分用电挪到便宜时段，一年能省多少钱？为了实现这件事购买的设备，大概几年能回本？

这个应用用比较直接的流程回答这个问题。第一步，用户编辑一天 24 小时的电价。电价编辑器支持负电价、自定义时段、新增时段、删除时段，并且会把每次修改整理成完整的 24 小时价格表。这样做很重要，因为后面的计算是按小时电价进行的。

第二步，用户选择家里有哪些可以灵活安排用电的设备。Demo 包含四类：储能电池、热泵热水、电动车充电和可预约家电。这几类都属于比较典型的、可以在时间上调整的家庭用电。

第三步，用户输入设备投入成本，也可以打开高级参数。高级参数包括电池往返效率、每天充放电次数、热泵性能系数，以及可选的按设备拆分成本。

点击计算后，应用会估算年节省和回本时间。结果页会展示主要回本数字、每年大概节省多少钱、各类设备分别贡献多少，以及简单的每日用电建议。它还可以生成 PNG 分享图片。后端可用时，当前状态和结果可以保存，并通过方案链接再次读取。

前端使用 React、TypeScript、Vite、Framer Motion 和 Lucide 图标。后端使用 Express。前端和后端都有同一套计算模型：API 不可用时前端可以本地计算，后端则提供 API 计算和基于文件的方案保存。

这个项目有意保持清楚，不声称能代替完整工程仿真。它没有模拟天气、真实家庭负荷曲线、光伏上网、电池详细衰减、税费、补贴或实时市场结算。它的作用是给用户一个实用的第一版估算，并把电价变化背后的计算逻辑展示出来。

简单来说，这个项目展示了怎样把一个真实的能源价格问题做成可使用的网页应用：有可编辑输入、有容易理解的结果、有后端 API、有方案保存、有分享图片、有自动测试，也有完全分开的中文和英文项目文档。

## 5 分钟版本

家庭用电回本计算器是一个移动端优先的网页应用，用于估算家庭在分时电价下可能获得的用电节省。项目从一个很直接的问题开始：当一天里的电价不断变化时，家庭能不能通过调整可移动用电，让储能电池、热泵或智能充电等设备的节省足以覆盖设备成本？

应用的第一部分是电价编辑器。用户可以设置一天里的负价、低价和高价。界面使用连续时段展示，因为这样更容易看懂；计算和编辑工具会把这些时段转换为 24 个小时价格。这样既能处理缺失和重叠，也能把价格相同的相邻小时重新合并。

第二部分是设备模型。应用不会假设所有家庭用电都能移动，而是提供四类明确设备。储能电池用低价电减少高价时段购电，并考虑往返效率；热需求通过热泵性能系数换算为电量；电动车按每周可安排的充电量计算；可预约家电则使用每天较小的可移动电量。

第三部分是经济结果。模型只计算正向电价差，并把每个已开启设备的节省折算成年节省。所有设备贡献汇总后，再用用户填写的投入成本除以总年节省。结果页先展示预计回本时间，再展示年节省、设备贡献明细，以及便宜和昂贵的用电时段。

前端负责全部用户体验，包括电价编辑、设备控制、本地状态保存、语言和货币格式、结果动画，以及 PNG 分享图片生成。后端提供健康检查、计算、保存方案和读取方案接口。后端无法连接时，浏览器会本地计算，所以 GitHub Pages 静态版本也可以使用。

这个计算结果有意定位为第一版估算。真正用于生产的能源规划工具，还需要真实电价、家庭测量数据、天气、光伏发电、设备功率限制、电池衰减、电网附加费、税费和当地市场规则。这个 Demo 不假装已经包含这些内容，它的目标是把主要取舍讲明白，同时展示一套完整的前后端产品流程。

## 常见问题回答

### 这是一个真实的财务计算器吗？

它是估算工具，适合用来理解电价差、可移动电量、年节省和回本时间之间的大致关系。它不是电费账单保证、工程建议或投资建议。

### 为什么前端也要做本地计算？

本地计算可以保证后端不可用时 Demo 仍能使用，包括部署到 GitHub Pages 这类静态托管的情况。后端仍然用于 API 计算和方案分享链接。

### 为什么前后端各有一套计算逻辑？

两套实现分别支持后端计算和静态页面兜底，测试会检查两边的重要行为。项目规模变大后，可以把共用模型提取到独立包中，降低重复维护风险。

### 这个项目最有价值的地方是什么？

它把真实的能源价格概念做成了一套完整、可操作的流程。电价编辑、结果明细、本地兜底、分享、多语言和自动测试，使它不只是一张静态效果图。

### 如果继续做成正式产品，下一步是什么？

下一步应当接入地区真实电价和家庭测量数据，然后增加用户账号、数据库、设备功率限制、光伏发电、电池衰减、税费、电网附加费和敏感性分析。

### 切换货币会自动换算金额吗？

不会。货币选项只改变符号和数字格式，不会请求实时汇率，也不会改变用户填写的金额数值。

### GitHub Pages 在线演示能保存方案吗？

不能。GitHub Pages 是静态托管，无法运行 Express API 或持久保存 JSON 文件。本地运行或部署到支持 Node.js 的服务器后，才能使用方案保存和恢复功能。
