# 项目展示流程

这份文档用于现场演示家庭用电回本计算器，适合课堂展示、作品集讲解、面试项目介绍或 GitHub 项目展示。完整流程大约需要五到八分钟；时间较短时，可以只讲项目简介、电价编辑、结果页和收尾总结。

## 1. 用一句话介绍项目

> 这个 Demo 用来估算一个家庭把可移动用电挪到低价时段后，一年大概能省多少钱，以及设备投入大概几年能回本。

## 2. 说明问题背景

很多用电方案已经不是全天一个固定价格。一天里可能出现凌晨便宜、白天正常、傍晚高峰昂贵的情况，有些市场甚至会在几个小时内出现负电价。

对拥有储能电池、热泵、电动车或可预约家电的家庭来说，这就带来一个实际机会：把一部分用电挪到便宜时段。这个计算器会根据电价差，粗略估算每年能节省的电费和设备回本时间。

建议讲法：

> 这个应用的核心是可移动用电。它不会假设家里所有负荷都能随便调整，只会计算用户实际选择的设备，并分别展示每类设备对结果的贡献。

## 3. 打开应用

公开的静态演示地址：

```text
https://jiexiaozhang1.github.io/energy-payback-calculator/
```

需要展示完整 Express 后端时，在本地运行生产版本：

```bash
npm install
npm run build
npm run start
```

然后打开：

```text
http://localhost:8787
```

需要前后端开发服务时运行：

```bash
npm run dev:full
```

GitHub Pages 属于静态托管，所以在线演示会使用浏览器本地计算。保存方案和根据分享链接恢复方案需要 Express 后端。

## 4. 介绍输入页

![中文输入页](assets/screenshot-input-zh.png)

按顺序介绍这些区域：

1. 顶部语言和货币控制。
2. 移动端优先的页面布局。
3. 一天 24 小时电价时间轴。
4. 四类可调度设备。
5. 设备总成本输入框。
6. 可以展开的高级参数。
7. 页面底部的计算按钮。

建议讲法：

> 第一页尽量做得直接，不要求用户先建立复杂的能源模型。用户只需要填电价、选择能调整用电时间的设备、填写投入成本，然后开始计算。

## 5. 演示电价编辑

![中文电价编辑器](assets/screenshot-tariff-editor-zh.png)

可以按下面的顺序操作：

1. 点击电价时间轴上的一个时段。
2. 展示从页面底部打开的编辑器。
3. 修改开始时间、结束时间或每 kWh 电价。
4. 说明输入框支持负电价。
5. 新增一个时段，或者删除现有时段。
6. 保存修改并回到完整时间轴。

需要说明的关键行为：

- 电价表始终覆盖 00:00 到 24:00。
- 每个时段至少覆盖一小时。
- 重叠或缺失的范围会通过按小时规范化来处理。
- 价格相同的相邻时段会自动合并。

建议讲法：

> 电价编辑器不只是几个输入框。每次修改都会被整理成一张干净、完整的 24 小时价格表，因为后面的节省计算依赖每个小时的电价。

## 6. 选择可调度设备

计算器包含四类设备：

1. 储能电池。
2. 热泵或热水负荷。
3. 电动车充电。
4. 可预约家电。

每类设备都可以单独开启或关闭。设备开启后，用户可以用滑块调整相关容量或可移动电量。

建议讲法：

> 用户只需要开启家里实际拥有的设备。系统会分别计算每类设备的年节省，因此结果页可以看出哪一类设备影响最大。

## 7. 展示成本和高级参数

默认流程只需要填写一个设备总成本。高级区域提供更多假设：

1. 电池往返效率。
2. 电池每天完整充放电次数。
3. 热泵性能系数。
4. 按已开启设备拆分成本。

建议讲法：

> 默认界面保持简单，高级参数则给了解设备效率和成本假设的用户更多控制空间。

## 8. 计算并介绍结果页

![中文结果页](assets/screenshot-result-zh.png)

使用示例参数后，点击“立即计算回本时间”。结果页可以从上往下介绍：

1. 预计回本时间。
2. 预计每年节省。
3. 结果来自后端还是前端本地计算。
4. 各类设备每年的节省贡献。
5. 每天便宜时段和昂贵时段的用电建议。
6. 生成分享图片的操作。
7. 返回修改输入的操作。

建议讲法：

> 结果页顶部直接回答最主要的问题，也就是投入大概多久能回本。下面的明细解释这些节省主要来自哪里，每日计划则把计算结果转成容易理解的用电建议。

不要把结果说成保证收益。它只是根据用户填写的电价、可移动电量、投入成本和效率参数做出的估算。

## 9. 解释前端和后端行为

面对技术类观众时，可以介绍下面的流程：

1. 前端把当前计算器状态发送到 `POST /api/calculate`。
2. 后端完成计算并返回结果。
3. 如果 API 不可用，前端会在浏览器中运行 `calculateSavings(state)`。
4. 两种情况下结果页都可以正常工作。
5. 保存方案时，前端把状态和结果发送到 `POST /api/scenarios`。
6. 后端在 `data/` 下写入自动生成的 JSON 文件。
7. 带有 `?scenario=<id>` 的网址可以恢复保存过的状态和结果。

建议讲法：

> 后端有意保持很小，主要用于展示 API 计算和方案保存。浏览器本地兜底则保证应用部署到静态托管或 API 暂时不可用时，核心计算仍然能使用。

## 10. 演示语言和货币控制

把界面在中文和英文之间切换，展示页面文案可以直接改变。项目还包含德语、法语和意大利语界面。

再切换货币显示，并明确说明：这里改变的是符号和数字格式，不会调用实时汇率转换。用户输入的金额数值本身不会改变。

## 11. 演示分享功能

在结果页使用分享操作：

1. 后端可用时，应用会尝试保存当前方案。
2. 应用准备一个包含方案编号的网址。
3. 浏览器支持剪贴板 API 时，会复制这个链接。
4. 应用使用 Canvas 生成一张 PNG 结果图片。
5. 如果后端保存失败，分享图片仍然可以继续生成。

这部分可以说明项目并不是算出一个数字就结束，而是把结果查看、保存和分享也做成了完整流程。

## 12. 收尾总结

> 家庭用电回本计算器把一个实际的分时电价问题做成了可以操作的网页应用。项目包含 24 小时电价编辑、可调度设备参数、年节省和回本估算、移动端优先界面、前端本地兜底、小型 Express API、方案保存、分享图片、自动测试，以及完全分开的中文和英文文档。
